# 安全政策

## 报告安全漏洞

我们非常重视安全问题。如果你发现了安全漏洞，**请勿通过公开的 GitHub Issue 报告**。

### 报告方式

1. **GitHub Security Advisory**（推荐）：通过 [GitHub Security Advisories](https://github.com/your-username/agent-platform/security/advisories/new) 提交
2. **邮件**：发送邮件至项目维护者（在 GitHub 个人资料中查找）

### 报告内容

请在报告中包含以下信息：

- 漏洞类型（如 SQL 注入、XSS、认证绕过等）
- 受影响的服务和版本
- 复现步骤
- 潜在影响
- 可能的修复建议（如有）

## 响应时间

| 阶段 | 时间目标 |
|------|----------|
| 确认收到报告 | 2 个工作日 |
| 初步评估 | 5 个工作日 |
| 修复方案确认 | 10 个工作日 |
| 发布修复 | 视严重程度而定 |

## 支持的版本

| 版本 | 支持状态 |
|------|----------|
| 最新发布版本 | ✅ 完全支持 |
| 前一版本 | 🔧 仅安全修复 |
| 更早版本 | ❌ 不再支持 |

## 安全最佳实践

部署本项目时，请注意以下安全事项：

1. **密钥管理**：所有密钥必须通过环境变量注入，禁止硬编码。参见 `.env.example`
2. **数据库密码**：生产环境必须使用强密码，不得使用 `dev_password` 等默认值
3. **JWT 密钥**：生产环境密钥长度至少 32 字符
4. **CORS 配置**：生产环境必须指定可信域名，禁止使用通配符
5. **mTLS**：生产环境服务间通信建议启用 mTLS
6. **审计日志**：确保 `audit_event` 表的触发器正常工作，防止删改
